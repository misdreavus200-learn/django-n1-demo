📖 N+1問題とは何か？

ORMを使った開発でよく発生するパフォーマンス問題です。
1回で済むはずのデータ取得が、N件のデータに対してN+1回のクエリに膨れ上がります。

1. 問題の仕組み

「書籍500件とその著者名・レビューを表示する」という処理を考えます。

❌ N+1が発生する場合 — 1,501本以上のSQL
Djangoアプリ → SELECT * FROM book（全件） → DB (1回目)
ループ×500回 → SELECT * FROM author WHERE id=1 → DB (2〜501回目)

＝ 合計 1 + 500（著者）+ 500×2（レビュー）= 1,501本以上のSQL
件数が増えるほどリニアに増加する

✅ 最適化した場合 — 2本のSQL
Djangoアプリ → SELECT book.*, author.* FROM book JOIN author …（全件） → DB (1回のみ)
Djangoアプリ → SELECT * FROM review WHERE book_id IN (…) → DB (2回目で完了)

＝ データが何件でも 3本のSQL（書籍＋著者JOIN / レビュー一括取得 / 集計）

2. コードの違い（たった1〜2行）

❌ N+1問題あり
# 著者情報は含まれていない
books = Book.objects.all()[:20]

for book in books:
    # ここで毎回SELECT発行
    name = book.author.name
    reviews = book.reviews.all()
✅ 最適化済み
# JOINと一括取得を指示する
books = Book.objects\
  .select_related('author')\
  .prefetch_related('reviews')\
  .all()[:20]

for book in books:
    name = book.author.name  # SQL不要
    reviews = book.reviews.all() # SQL不要
📌 使い分けのポイント
select_related … ForeignKey / OneToOne のとき（SQL JOINで取得）
prefetch_related … ManyToMany / reverse FK のとき（別クエリで一括取得・Pythonで結合）

3. なぜ気づきにくいのか？

Djangoのquerysetは 遅延評価（Lazy Loading） が基本です。
book.author.name のようにアクセスした瞬間にSQLが発行されるため、 コードを読んだだけでは問題に気づきにくいのです。

  • 少ないデータ件数の開発環境では遅さを感じない
  • 本番環境でデータが増えて突然パフォーマンスが劣化する
  • テンプレート内で {{ book.author.name }} と書くだけでも発生する

4. 「DB時間以外」でも遅くなる理由

このサイトで計測している 「DB合計時間」 は、SQLiteがSQLを実際に実行した時間だけです。
しかし画面表示時間はDB時間よりもはるかに大きくなります。 実際の計測例を見てみましょう。

📊 実測値の例（書籍500件）
計測項目 ❌ N+1あり ✅ 最適化済み
DB合計時間 397 ms 1 ms
DB以外の処理時間 2,980 ms 736 ms
画面表示時間（合計） 3,377 ms 737 ms

※ DB以外の時間差（2,244ms）はSQLの本数が原因で発生するPython/ORMのオーバーヘッドです

1本のクエリを発行するたびに、SQLの実行時間とは別に以下の処理が発生します。

1クエリあたりの処理コスト（N+1版では1,500回繰り返される）
① ORM → SQLを文字列として組み立てる
② ドライバ → SQLiteへのシステムコール発行
③ SQLite → SQL実行・結果をカーソルに格納 ← DB時間として計測されるのはここだけ
④ ORM → 結果行をPythonのModelオブジェクトに変換
⑤ Python → 次のループ処理へ

①②④⑤ の処理が1,500回積み重なる → DB時間とは別に数千msのオーバーヘッドが発生する

📌 N+1問題の本質
「SQLが1本あたり遅い」のではなく、「SQLを呼ぶ回数が多すぎる」ことがボトルネックです。
DB時間だけを見ていると問題の大きさを過小評価してしまいます。 画面表示時間全体で比較することが重要です。

5. 検出ツール

🔧 django-debug-toolbar

開発中にブラウザのサイドパネルで発行SQLを全て確認できる。重複クエリも警告してくれる。

🐍 django-silk

本番に近い環境でのプロファイリングに向いている。リクエストごとのSQL数・時間を記録。

📊 connection.queries

from django.db import connection で全クエリのリストをPythonコードから取得できる。

🧪 nplusone

N+1問題をテストコードで検出できるライブラリ。CI/CDに組み込んでの予防に最適。
