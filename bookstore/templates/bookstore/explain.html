📖 N+1問題とは何か？

ORMを使った開発でよく発生するパフォーマンス問題です。
1回で済むはずのデータ取得が、N件のデータに対してN+1回のクエリに膨れ上がります。

1. 問題の仕組み

「書籍20件とその著者名を表示する」という処理を考えます。

❌ N+1が発生する場合 — 21本のSQL
Djangoアプリ → SELECT * FROM book LIMIT 20 → DB (1回目)
ループ×20回 → SELECT * FROM author WHERE id=1 → DB (2〜21回目)

＝ 合計 1 + 20 = 21本のSQL（データが100件なら101本！）

✅ 最適化した場合 — 2本のSQL
Djangoアプリ → SELECT book.*, author.* FROM book JOIN author … LIMIT 20 → DB (1回のみ)
Djangoアプリ → SELECT * FROM review WHERE book_id IN (…) → DB (2回目で完了)

＝ データが何件でも 2本のSQL

2. コードの違い（たった1〜2行）

❌ N+1問題あり
# 著者情報は含まれていない
books = Book.objects.all()[:20]

for book in books:
    # ここで毎回SELECT発行
    name = book.author.name
    reviews = book.reviews.all()
✅ 最適化済み
# JOINと一括取得を指示する
books = Book.objects\
  .select_related('author')\
  .prefetch_related('reviews')\
  .all()[:20]

for book in books:
    name = book.author.name  # SQL不要
    reviews = book.reviews.all() # SQL不要
📌 使い分けのポイント
select_related … ForeignKey / OneToOne のとき（SQL JOINで取得）
prefetch_related … ManyToMany / reverse FK のとき（別クエリで一括取得・Pythonで結合）

3. なぜ気づきにくいのか？

Djangoのquerysetは 遅延評価（Lazy Loading） が基本です。
book.author.name のようにアクセスした瞬間にSQLが発行されるため、 コードを読んだだけでは問題に気づきにくいのです。

  • 少ないデータ件数の開発環境では遅さを感じない
  • 本番環境でデータが増えて突然パフォーマンスが劣化する
  • テンプレート内で {{ book.author.name }} と書くだけでも発生する

4. 検出ツール

🔧 django-debug-toolbar

開発中にブラウザのサイドパネルで発行SQLを全て確認できる。重複クエリも警告してくれる。

🐍 django-silk

本番に近い環境でのプロファイリングに向いている。リクエストごとのSQL数・時間を記録。

📊 connection.queries

from django.db import connection で全クエリのリストをPythonコードから取得できる。

🧪 nplusone

N+1問題をテストコードで検出できるライブラリ。CI/CDに組み込んでの予防に最適。
